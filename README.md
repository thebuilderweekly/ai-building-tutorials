# Builder Weekly Tutorials

Agent-building patterns. Each entry teaches one task where an AI agent measurably beats a human on speed, consistency, or reliability.

Every published entry contains three things:

1. A **before-state** describing what the task looks like without the agent.
2. An **after-state** describing what the task looks like with the agent.
3. One sentence — the **soul line** — naming explicitly why an agent beats a human at this specific task.

The corpus is the source of truth for [thebuilderweekly.com/tutorials](https://thebuilderweekly.com/tutorials). Contributions welcome — see [CONTRIBUTING.md](./CONTRIBUTING.md).

Operational entries that don't fit a single pillar (competitor pricing watch, GitHub issue triage, RSS filtering, etc.) use the `operational` pillar tag and appear on the site grouped separately.

## The bar

A tutorial earns a place in this corpus if it teaches **one specific task** where an agent measurably beats a human — on speed, on consistency, on reliability, or on cost. "Measurably" is the load-bearing word. If the improvement isn't numerically defensible (40x speed, 100% recall across N runs, zero variance over M days), the entry doesn't pass the merge gate.

Read the full inclusion criterion and the seven-check merge gate in [MERGE_GATE.md](./MERGE_GATE.md).

## How entries are produced

Every entry is generated by an autonomous pipeline, verified by an eight-check rubric, and approved by a human before publishing. The same automated merge gate applies to community contributions — there are no exceptions for the in-house pipeline versus an external PR.

The production queue lives at [`queue/topics.json`](./queue/topics.json). Entries are picked in priority order.
